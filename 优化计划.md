# 农产品供销系统 - 优化计划

## 背景

本计划基于当前项目已完成的几轮修复继续整理，目标是把后续工作拆成可验证、低风险的小阶段。

当前项目已经完成过鉴权、密码哈希、上传限制、订单链路、删除保护、本地配置文件等修复，因此本计划不再把所有问题当成“从零开始”。后续实施应遵守三条原则：

- 先修确定存在、范围小、容易验证的问题。
- 数据库结构调整单独处理，必须写迁移脚本，不直接重建库。
- 大重构后置，避免破坏刚修好的权限、校验和订单状态逻辑。

每个阶段完成后都至少运行：

```powershell
cd springboot
mvn -q -DskipTests compile

cd ..\vue
npm run build
```

涉及接口行为的阶段，还要手动验证登录、下单、订单状态、上传、删除保护。

## 第三阶段：数据库迁移计划（单独分支或单独提交）

> 目标：修正数据库结构问题，但必须先保护已有数据。

### 3.1 迁移前检查

**文件：** 新建迁移检查 SQL，例如 `mysql/migrations/001_precheck.sql`

先检查：

- [ ] `orders.goods_id` 是否全部能转换为整数
- [ ] `notice.time`、`orders.time` 是否全部能转换为 `DATETIME`
- [ ] `goods_stock.date` 是否全部能转换为 `DATE`
- [ ] 外键候选数据是否存在孤儿记录，例如订单里的 `goods_id` 是否都能在 `goods` 表找到
- [ ] 关键字段是否存在 NULL 或空字符串

### 3.2 Schema 迁移

**文件：** 新建迁移 SQL，例如 `mysql/migrations/002_schema_cleanup.sql`

确认预检查通过后再做：

- [ ] `orders.goods_id` 从 `varchar(255)` 改为 `int(11)`
- [ ] `notice.time` 改为 `DATETIME`
- [ ] `orders.time` 改为 `DATETIME`
- [ ] `goods_stock.date` 改为 `DATE`
- [ ] 关键字段逐步添加 `NOT NULL` 约束
- [ ] 添加索引：`goods(category_id)`、`orders(goods_id)`、`orders(user_id)`、`orders(status)`、`goods_stock(goods_id)`
- [ ] 外键约束后置，只有在确认无历史脏数据且业务删除规则稳定后再加

### 3.3 初始化脚本同步

**文件：** `mysql/farm_system.sql`

- [ ] 迁移脚本验证通过后，再同步更新初始化 SQL
- [ ] 不用直接重建数据库替代迁移
- [ ] 验证：新库可从初始化 SQL 导入，旧库可通过迁移脚本升级

### 3.4 时间戳字段

`created_at`、`updated_at` 有价值，但会影响实体类、Mapper、初始化数据和前端展示。

- [ ] 暂不和类型修复混做
- [ ] 作为独立数据库版本处理

## 第四阶段：后端代码质量优化（谨慎推进）

> 目标：补齐事务、校验、测试，暂缓大范围抽象。

### 4.1 事务补齐

**文件：** 各 Service

- [ ] 写操作按实际需要添加 `@Transactional`
- [ ] DB 写入和 Redis 缓存失效在同一事务边界内处理
- [ ] 查询接口需要时再添加 `@Transactional(readOnly = true)`
- [ ] 验证：新增、修改、删除、订单状态变更仍正常

### 4.2 输入校验补齐

当前项目已有一轮后端手写校验，不应描述为完全缺失。

建议：

- [ ] 先补页面入口最关键的校验：注册、登录、改密、商品价格、库存、订单数量
- [ ] 再考虑引入 `spring-boot-starter-validation`
- [ ] 如果引入 JSR-303，先从少量 DTO 开始，不直接给所有 Entity 大面积加注解
- [ ] `selectPage` 增加 `pageSize` 上限，建议最大 100

### 4.3 API 规范化

这些改动牵动前后端同步，后置处理：

- [ ] `GET /logout` 改为 `POST /logout`
- [ ] `Result.code` 从 `String` 改为 `Integer`
- [ ] 404 资源不存在时返回 HTTP 404

执行要求：

- 单独提交
- 同步改前端 request 逻辑
- 保留兼容说明或迁移说明

### 4.4 依赖和包名整理

- [ ] `java-jwt` 升级到 4.x 前先确认 API 兼容
- [ ] 添加 `spring-boot-starter-test` 可优先做
- [ ] 添加 `<project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>`
- [ ] `com.example.Utils` 改为 `com.example.utils` 属于包名重命名，后置到独立提交
- [ ] 删除“缺少 Lombok 依赖”的说法；当前项目未使用 Lombok，不建议为了简化实体类引入 Lombok

### 4.5 暂缓的大重构

以下内容先不作为近期任务：

- 泛型 `BaseService`
- 泛型 `BaseCrudController`
- `AdminService`、`UserService` 抽公共基类
- `CategoryService`、`NoticeService` 抽缓存基类

原因：

- 当前优先级不如安全、数据和权限问题高
- 一次性重构范围大，容易引入回归
- 等测试补起来后再做更稳

## 推荐实施顺序

| 顺序 | 阶段 | 风险 | 说明 |
| --- | --- | --- | --- |
| 1 | 第一阶段：小范围确定性修复 | 低到中 | 最容易验证，优先收益高 |
| 2 | 第二阶段：安全配置加固 | 中 | 需要兼顾本地启动体验 |
| 3 | 第三阶段：数据库迁移计划 | 高 | 必须写预检查和迁移 SQL |
| 4 | 第四阶段：后端代码质量优化 | 中 | 先补事务、校验、测试，暂缓大重构 |
| 5 | 第五阶段：前端体验和结构优化 | 中 | 功能稳定后逐步做 |
| 6 | 第六阶段：测试和长期维护 | 低到中 | 为后续重构提供保护 |

## 当前不建议马上做的事

- 不建议直接重建数据库来“修复” Schema。
- 不建议马上引入泛型 Controller/Service。
- 不建议一次性引入 Pinia 并重构所有页面。
- 不建议直接把 `/files/download` 全部改成登录后可访问。
- 不建议把 `Result.code` 类型修改混进安全修复提交。
- 不建议为了实体类简写引入 Lombok。

## 下一步建议

下一步从第一阶段开始，建议单独提交：

1. 修 `Password.vue` 拼写和小范围前端死代码。
2. 修 `StatisticsService` 计数性能问题。
3. 补 `OrdersService` 的订单归属和删除规则。
4. 收敛文件下载目录。

每完成一项都运行后端编译和前端构建，涉及接口的项再做手动功能验证。
