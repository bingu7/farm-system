# Codex 使用教程

> 适用对象：第一次使用 Codex，或希望把 Codex 当作项目协作助手来使用的人。
>
> 说明：本文基于当前 Codex 桌面环境和常见工作流整理。生成时尝试获取 OpenAI Codex 官方手册，但远程访问返回 403，因此涉及产品界面和功能名称的内容以后可能需要按最新版官方文档核对。

## 1. Codex 是什么

Codex 是一个面向软件开发和文件处理的 AI 协作助手。它可以在你的项目目录中读取文件、分析代码、修改文件、运行命令、启动服务、检查报错、生成文档，并把每一步结果反馈给你。

你可以把它理解成一个会看项目、会执行命令、会写代码的协作者。它不是只给建议，也可以直接在本机工作区里完成任务。

## 2. Codex 适合做什么

Codex 适合处理这些任务：

- 阅读项目结构，解释代码逻辑。
- 修复 bug，并运行测试或构建命令验证结果。
- 新增功能、接口、页面、配置项。
- 生成 README、接口文档、使用说明、交接文档。
- 检查 Git 改动，整理提交说明。
- 调试前后端启动失败、依赖问题、编译错误。
- 批量处理项目文件，但应先明确范围。

不建议直接让 Codex 做这些事：

- 没有说明边界的大规模重构。
- 删除文件、重置 Git、覆盖重要数据。
- 需要账号密码、隐私数据、线上生产权限的操作。
- 没有验证条件的模糊请求，比如“把项目优化一下”。

## 3. 基本使用流程

推荐使用下面的流程和 Codex 协作：

1. 说明目标。
2. 说明范围。
3. 说明成功标准。
4. 让 Codex 执行。
5. 让 Codex 验证。
6. 查看结果，再决定是否继续。

示例：

```text
帮我修复登录接口报错。
范围只限 springboot 后端，不改前端。
成功标准：项目能编译通过，并说明改了哪些文件。
```

再比如：

```text
给这个项目生成一份接口文档，放在根目录。
只根据现有 controller 和 service 写，不要编造接口。
生成后帮我检查文件是否存在。
```

## 4. 如何写好提示词

好的提示词应该包含四个信息：

- 目标：你想完成什么。
- 范围：哪些文件、模块、页面可以改。
- 限制：哪些东西不能动。
- 验证：完成后用什么方式确认。

推荐模板：

```text
请帮我完成：[目标]
修改范围：[文件/目录/模块]
不要修改：[禁止改动的内容]
完成标准：[构建通过/测试通过/页面可访问/文档生成成功]
最后请告诉我：[修改文件、验证结果、注意事项]
```

例子：

```text
请帮我修复商品图片上传失败的问题。
修改范围：springboot 后端上传相关代码和 vue 前端上传组件。
不要改数据库表结构。
完成标准：后端编译通过，前端 build 通过。
最后请告诉我改了哪些文件，以及是否还有风险。
```

## 5. 让 Codex 修改代码

当你希望 Codex 直接改代码时，最好说清楚是否需要“先计划”。

如果你还不确定方案，可以这样说：

```text
先不要改代码，先帮我分析原因并给出修改计划。
```

如果你已经确定要它执行，可以这样说：

```text
按最小改动修复这个问题，修完后运行测试验证。
```

Codex 修改代码时通常会先读取相关文件，再做小范围修改。你可以要求它保持改动克制：

```text
只改和这个 bug 直接相关的代码，不做额外重构。
```

## 6. 让 Codex 运行命令

Codex 可以运行本机命令，例如：

```text
帮我运行后端编译命令，并解释报错。
```

```text
帮我启动前端开发服务器，告诉我访问地址。
```

常见命令包括：

```bash
npm run build
mvn -q -DskipTests compile
mvn spring-boot:run
git status
git diff
```

如果命令需要联网、写入受限目录、启动 GUI、执行危险操作，Codex 可能会向你申请权限。你需要根据操作风险决定是否允许。

## 7. 权限和安全

使用 Codex 时要特别注意权限边界：

- 普通读写：通常只能在当前工作区内进行。
- 联网操作：可能需要你的批准。
- 删除、重置、覆盖：应谨慎，最好先备份。
- Git 操作：提交、推送、切换分支前最好确认。
- 敏感信息：不要把真实密码、Token、身份证、银行卡等直接发给 Codex。

如果你担心某个操作风险高，可以先让 Codex 只分析：

```text
先不要执行，告诉我这个命令会做什么，有没有风险。
```

## 8. AGENTS.md / CLAUDE.md 的作用

项目里的 `AGENTS.md` 或类似说明文件，可以用来告诉 Codex 在这个仓库中应该遵守什么规则。

适合写进去的内容：

- 项目启动命令。
- 测试和构建命令。
- 代码风格要求。
- 哪些目录不能随便改。
- 修改前是否必须先给计划。
- 提交信息格式。
- 安全注意事项。

例如当前项目中的规则强调：

- 不要盲目假设。
- 优先简单方案。
- 只做和任务直接相关的修改。
- 多步骤任务要有明确验证标准。

这类规则能减少误改、过度重构和无关变更。

## 9. 常见协作场景

### 9.1 解释项目

```text
帮我快速了解这个项目结构，说明前端、后端、数据库分别在哪里。
```

### 9.2 修复报错

```text
后端启动失败了，帮我看日志并定位原因。
只分析和启动失败相关的内容。
```

### 9.3 新增功能

```text
帮我新增商品分类管理功能。
先给我计划，包括要改哪些文件、接口、页面和验证方式。
等我确认后再开始改。
```

### 9.4 生成文档

```text
根据现有代码生成接口文档，保存为 接口文档.md。
不要编造不存在的接口。
生成后帮我检查文档是否包含主要模块。
```

### 9.5 做代码审查

```text
请 review 当前改动，优先找 bug、风险和缺少测试的地方。
不要直接修改代码。
```

### 9.6 提交 Git

```text
帮我查看当前 git diff，总结改动。
确认没有问题后，帮我提交，提交信息用中文。
```

## 10. 常见问题

### Codex 会不会自动改我的文件？

只有当你的请求需要修改文件，或者你明确让它执行任务时，它才会改。你也可以明确说“不要修改文件，只分析”。

### Codex 改错了怎么办？

可以让它查看 `git diff`，解释每个改动，并按你的要求继续修正。不要轻易使用强制重置命令，除非你确认要丢弃改动。

### Codex 能不能联网？

取决于当前环境和权限设置。联网查询、下载依赖、访问外部文档等操作可能需要你批准。

### Codex 能不能保证结果正确？

不能只靠口头保证。重要任务应要求它运行测试、构建、页面检查或文件校验。没有验证过的结果，只能视为初步完成。

### 什么情况下应该先让 Codex 做计划？

以下情况建议先计划：

- 要改多个模块。
- 要动数据库、权限、支付、订单等关键流程。
- 你还不确定需求边界。
- 项目已有很多历史改动，担心误伤。

## 12. 一套稳妥的日常工作流

可以按这个顺序使用 Codex：

1. 让 Codex 读项目结构。
2. 说明你要解决的问题。
3. 让 Codex 给出计划。
4. 确认计划后再让它修改。
5. 修改后运行构建、测试或启动验证。
6. 查看 `git diff`。
7. 让 Codex 总结改动和风险。
8. 确认无误后再提交。

示例完整指令：

```text
帮我修复订单提交失败的问题。
先分析，不要改代码。
重点看 springboot 后端订单相关 controller、service、mapper，以及前端订单提交页面。
请给出原因、修改计划和验证方式。
```

确认方案后：

```text
按刚才的计划开始修改。
要求最小改动，不做无关重构。
修完后运行后端编译，并告诉我修改文件和验证结果。
```

## 13. 使用建议

把 Codex 当成一个执行力很强的协作者，而不是一次性许愿机。越清楚地说明目标、范围和验证方式，它越容易给出可靠结果。

最重要的三句话：

```text
先分析，再修改。
只改必要内容。
修完必须验证。
```
