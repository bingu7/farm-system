# 鉴权与密码体系修复笔记

这份笔记对应本次对项目“鉴权与密码体系”的第一轮修复，目的是方便后续自己复盘、继续优化。

## 1. 原来主要有什么问题

### 1.1 密码是明文流转的

- 数据库里直接存明文密码
- 登录成功后，后端把带 `password` 的用户对象直接返回给前端
- 前端又把整个对象塞进了 `localStorage`
- Redis 登录态里也缓存了完整用户对象
- JWT 签名还直接用了用户密码

这会带来两个问题：

1. 安全性差，密码暴露面太大
2. 一旦改密码，token、缓存、前端本地数据都会互相牵扯

### 1.2 只有“登录校验”，没有“权限校验”

原来的 `JwtInterceptor` 只是验证有没有 token，并没有真正限制：

- 普通用户能不能调用管理员接口
- 用户能不能改别人的数据
- 统计接口能不能随便传别人的 `userId`

所以那时候更像是“有登录态”，但不是“有权限系统”。

## 2. 这次修了什么

## 2.1 密码改成哈希存储

这次新增了：

- `spring-security-crypto`
- `PasswordUtils`

目前用的是 `BCryptPasswordEncoder`，这是 Spring 体系里很常见的密码哈希方式。

现在的规则是：

- 新增管理员/用户时，密码先哈希再入库
- 修改密码时，密码先哈希再入库
- 登录时使用 `matches(raw, hashed)` 比对

## 2.2 保留了“旧明文密码”的兼容迁移

这是这次一个比较重要的处理。

因为你现在的 SQL 初始化数据和已有数据库里，很多密码还是旧的明文。如果我这次直接强切成“只认 BCrypt”，你原来的演示账号会立刻全登不上。

所以我做了一个兼容策略：

- 如果数据库里的密码已经是 BCrypt 哈希，就按 BCrypt 校验
- 如果数据库里的密码还是旧明文，就先按旧逻辑校验
- 只要旧明文登录成功，就立刻自动升级成 BCrypt 哈希再写回数据库

这叫“登录时迁移”。

好处是：

- 不需要你手工先改数据库
- 老账号还能继续用
- 账号只要成功登录一次，就自动升级成新体系

## 2.3 登录响应不再下发密码

这次加了 `AccountSanitizer`，它会在对外返回前把这些敏感字段清掉：

- `password`
- `newPassword`

所以现在：

- 登录响应不会再把密码发给前端
- 用户列表、管理员列表、按 ID 查询，也不会再把密码带回去

这一步非常重要，因为以前就算数据库改成哈希，只要接口还把 `password` 返回给前端，本质上还是把敏感信息往外送。

## 2.4 JWT 不再使用“用户密码作为签名密钥”

原来 `TokenUtils` 是这样设计的：

- 每个用户 token 的签名 secret = 用户自己的密码

这会导致几个问题：

- 密码和 token 体系强耦合
- 改密码后旧 token 的行为更难控
- Redis 和 JWT 校验都必须依赖密码字段

现在改成：

- JWT 统一使用服务端 secret 签名
- 默认值是代码里的 `farm-system-jwt-secret`
- 如果以后部署，可以通过环境变量 `JWT_SECRET` 覆盖

这样就把“密码校验”和“token 签发”拆开了。

## 2.5 加了当前登录用户上下文

这次新增了：

- `LoginUserHolder`
- `AuthUtils`

思路是：

1. `JwtInterceptor` 拦截请求
2. 校验 JWT
3. 从 Redis 拿到当前登录用户
4. 放进 `ThreadLocal`
5. 控制器和业务代码里通过 `AuthUtils.getCurrentUser()` 拿当前用户

这样以后很多接口就不用再信任前端传过来的 `userId`、`role` 了。

## 2.6 加了角色/归属校验

这次先补的是项目里最关键的一层权限约束。

### 管理员接口

管理员接口现在会要求 `ADMIN` 角色，例如：

- `/admin/**`
- `/user/selectPage`
- `/user/add`
- `/user/delete`
- 商品、分类、进货、公告的新增/修改/删除

### 用户归属接口

和“当前登录用户自己”相关的接口，现在会做归属校验，例如：

- 用户修改自己的资料
- 查看自己的订单
- 修改自己的订单状态
- 统计只看自己的数据

核心原则是：

- 管理员可以管理全局
- 普通用户只能操作属于自己的数据

## 2.7 统计接口不再信任前端传参

原来的 `/statistics` 是前端传：

- `userId`
- `role`

这其实很危险，因为任何人都能改请求参数。

现在改成：

- 后端直接从当前登录态里取 `id` 和 `role`
- 前端不再传这两个参数

这个改法以后也适合你推广到别的接口上。

## 2.8 修改密码后，当前 token 会立即失效

这次在 `/updatePassword` 里增加了一个处理：

- 改完密码后，删除当前 token 在 Redis 里的登录态

这样前端清掉本地缓存并跳回登录页时，服务端这边也同步失效，不会出现“前端退出了，但旧 token 其实还能继续用”的问题。

## 2.9 顺手修了一个和权限有关的订单风险

这次虽然主目标不是订单，但因为做“归属校验”时刚好碰到一个明显漏洞，所以一起收了：

- 原来 `/orders/update` 会直接信任前端传来的 `goodsId`、`num`
- 取消订单返库存时也直接用前端传入值

现在改成：

- 先根据订单 ID 查数据库
- 返库存时只使用数据库中的订单数据
- 只更新允许变动的状态字段

这属于“不要信任前端传来的业务关键字段”。

## 3. 这次修改后，你要注意什么

### 3.1 旧数据库里的演示账号第一次登录时会自动升级

也就是说：

- 你第一次用旧账号登录，表里还是明文
- 登录成功后，系统会自动把这条密码改成 BCrypt 哈希

这不是 bug，是这次为了平滑迁移故意保留的策略。

### 3.2 前端本地缓存里不再有 password

所以以后你再做前端功能时，不要默认 `localStorage` 里的 `system-user` 一定有 `password`。

正确思路应该是：

- 本地缓存只存“显示和鉴权必需的信息”
- 密码只在用户输入时短暂存在表单里

## 4. 这次还没有做到的事情

这次是第一轮修复，还不是最终版。后面继续进阶，建议按这个顺序做：

### 4.1 文件上传也做登录校验和类型限制

当前 `/files/upload` 仍然偏开放，建议后面补：

- 只允许登录后上传
- 限制图片类型
- 服务端生成安全文件名

### 4.2 订单状态流转再严格一点

现在只是把“谁能改谁的订单”和“返库存不能信前端”补了。

后面还可以继续收紧：

- 用户只能 `待支付 -> 已取消 / 待发货`
- 管理员只能 `待发货 -> 待收货`
- 用户只能 `待收货 -> 已完成`

### 4.3 初始化 SQL 也已经切成哈希密码

这次顺手把初始化脚本里的演示密码也换成了 BCrypt 哈希值。

这意味着：

- 新建数据库后，演示账号默认就是哈希存储
- 老数据库依然可以走“登录时自动升级”的兼容路线
- 新旧库都能平滑使用

### 4.4 再往后可以考虑 Spring Security

如果项目以后要继续扩展，功能变复杂，可以考虑引入 Spring Security。

但对你当前这个项目来说，这次做的这套“轻量版权限体系”已经足够比原来安全、清晰很多，而且更适合你现在学习。

## 5. 这次最值得记住的几个原则

### 原则 1：密码永远不要明文存储

正确做法不是“把密码取回来比一下”，而是：

- 入库前哈希
- 登录时用 `matches`

### 原则 2：不要把密码返回给前端

哪怕已经是哈希值，也不应该随便下发给前端或放进本地缓存。

### 原则 3：不要信任前端传来的身份信息

像 `userId`、`role`、`goodsId`、`num` 这种关键字段，能以后端登录态和数据库为准，就不要以前端参数为准。

### 原则 4：登录校验不等于权限校验

“你是谁”和“你能做什么”是两回事。

有 token 只能说明“你登录了”，不能说明“你有管理员权限”或者“这个订单就是你的”。

### 原则 5：迁移旧系统时，兼容策略很重要

很多时候不能一步切死旧逻辑，否则线上或演示环境会立刻坏掉。

这次“旧明文登录成功后自动升级”为哈希，就是一个典型的过渡方案。
