# 文件上传鉴权与类型限制修复笔记

这份笔记对应本次对“文件上传链路”的修复，重点是两件事：

1. 只有登录用户才能上传
2. 只允许上传系统真正需要的图片类型

## 1. 原来的问题在哪里

### 1.1 `/files/**` 整段都被放行了

原来在 `WebConfig` 里，拦截器把 `/files/**` 全部排除了。

这意味着：

- 不登录也能上传文件
- 只要知道接口地址，任何人都可以往服务端磁盘写文件

这和前面刚修好的登录态、权限体系是断开的。

## 1.2 服务端信任了用户传来的原始文件名

原来的上传逻辑是：

- 直接拿 `originalFilename`
- 前面拼一个时间戳
- 然后保存到本地目录

这个做法的问题是：

- 文件名里可能带奇怪字符
- 文件名可能很长，不稳定
- 服务端文件命名规则被客户端输入牵着走

更稳妥的做法是：服务端自己生成文件名，只保留经过校验的扩展名。

## 1.3 没有限制文件类型

原来接口几乎是“来什么收什么”。

但你这个项目里上传场景其实很明确：

- 用户头像
- 管理员头像
- 商品图片

所以真正需要的只是图片文件，而不是任意文件。

如果不限制类型，会带来两个现实问题：

1. 业务上没必要，上传面太宽
2. 安全上不合适，容易把脚本、压缩包、可执行文件之类的东西塞进来

## 2. 这次怎么修的

## 2.1 只放行下载，不再放行上传

现在 `WebConfig` 改成了：

- `/files/download/**` 继续允许匿名访问
- `/files/upload` 不再排除，必须经过 `JwtInterceptor`

这样设计的原因是：

- 图片下载常常要给页面直接展示
- 上传属于“修改服务器状态”的动作，必须和登录态绑定

一句话概括：

- 下载可以公开
- 上传必须带 token

## 2.2 前端上传组件补上 token

普通 `axios` 请求会自动走你已经写好的请求拦截器，但 `el-upload` 是单独发请求的，不会自动继承 `request.js` 里的 token 注入逻辑。

所以这次专门加了一个上传工具：

- `vue/src/utils/upload.js`

它做了两件事：

1. 从 `localStorage` 里取当前登录用户的 token
2. 作为 `headers` 传给 `el-upload`

这一步很关键，因为很多人第一次做 Element Plus 上传时，都会以为“既然页面已经登录了，上传自然也会带 token”，其实不会。

## 2.3 服务端限制图片类型

这次后端只允许这些扩展名：

- `jpg`
- `jpeg`
- `png`
- `gif`
- `webp`
- `bmp`

同时也校验了 `contentType`，只接受常见图片 MIME。

这样做的思路是“双保险”：

- 扩展名校验，拦住最直接的不合法文件名
- MIME 校验，再补一层服务端判断

要注意一点：

MIME 不能 100% 代表文件内容绝对安全，但对你现在这个学习项目来说，这已经比原来的“完全不限制”强很多，而且复杂度也合适。

## 2.4 服务端自己生成文件名

现在上传时不再使用用户原始文件名保存，而是：

- 服务端生成 UUID
- 只拼接通过校验的扩展名

例如会变成这种风格：

- `7d9c4f0b4f404e3b9f5a2f3c0d1e8a12.png`

这样做的好处是：

- 文件名规则统一
- 不容易碰撞
- 不受用户原始文件名污染

## 2.5 下载也顺手补了路径安全

虽然这次用户需求主要是“上传鉴权和类型限制”，但下载接口原来也是直接拼路径。

所以这次顺手补了两层保护：

1. 下载文件扩展名也必须在允许列表里
2. 最终路径必须仍然落在 `files` 目录下

这类处理的核心原则是：

不要把路径当字符串随便拼，要把它当受约束的文件系统对象去校验。

## 2.6 前端也补了一个轻量的类型提示

除了后端真校验，这次前端也在上传前加了 `before-upload`：

- 非图片文件会先在浏览器里被拦住
- 用户会更早看到提示

这不是安全边界，真正的安全边界仍然是后端。

但它能减少误操作，体验会好一些。

## 3. 这次改动涉及哪些文件

后端：

- `springboot/src/main/java/com/example/config/WebConfig.java`
- `springboot/src/main/java/com/example/controller/FileController.java`

前端：

- `vue/src/utils/upload.js`
- `vue/src/views/manager/Person.vue`
- `vue/src/views/manager/Admin.vue`
- `vue/src/views/manager/User.vue`
- `vue/src/views/manager/Goods.vue`

## 4. 这次最值得记住的几个点

### 原则 1：文件上传本质上是“写服务器”

只要是写服务器，就不要因为“它只是传图片”而放松鉴权。

## 原则 2：前端校验只是体验，后端校验才是边界

前端 `before-upload` 可以拦误操作，但真正决定“能不能收”的是后端。

## 原则 3：不要信任用户给你的文件名

文件名、路径、扩展名，这些都应该由服务端掌控。

## 原则 4：白名单比黑名单更适合上传接口

你这个系统只需要图片，那就明确只允许图片。

不要去想“禁止 exe、禁止 zip、禁止 bat 是否够了”，因为黑名单永远容易漏。

## 5. 后面还能继续怎么优化

这次修的是一个很合适的第一版，后面如果你继续打磨，可以再往下走：

### 5.1 加文件大小限制

比如头像限制 2MB，商品图限制 5MB。

这样可以避免超大图拖慢上传和页面展示。

### 5.2 区分上传用途

现在头像和商品图共用一个上传接口。后面如果你想更细，可以拆成：

- `/files/upload/avatar`
- `/files/upload/goods`

然后分别限制尺寸、大小、角色权限。

### 5.3 把文件元数据存数据库

现在只是把图片 URL 存在业务表里。以后也可以单独建文件表，记录：

- 上传人
- 上传时间
- 原始文件名
- 实际存储名
- 文件类型

这样会更利于管理和审计。

### 5.4 上生产时再考虑对象存储

如果以后不只是学习项目，而是要长期部署，可以考虑把本地磁盘上传改成：

- OSS
- COS
- MinIO

但对你当前这个项目，先把“登录鉴权 + 类型限制 + 文件名安全”做好，优先级是完全正确的。
