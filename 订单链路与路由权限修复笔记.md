# 订单链路与路由权限修复笔记

这份笔记记录本次对“第一优先级问题”的修复和测试结果，主要覆盖三块：

1. 订单状态流转限制
2. 库存并发扣减
3. 前端路由守卫

## 1. 为什么要修

### 1.1 订单状态不能只靠前端按钮限制

原来前端会根据角色和订单状态显示不同按钮，比如：

- 用户在待支付时可以支付或取消
- 管理员在待发货时可以发货
- 用户在待收货时可以确认收货

但前端按钮只是界面限制，不能当成真正的业务安全边界。

如果有人绕过页面，直接调用 `/orders/update` 接口，仍然可能传入任意 `status`。所以状态流转规则必须放在后端。

### 1.2 下单扣库存有并发风险

原来的逻辑大致是：

1. 查询商品库存
2. Java 里计算 `store - num`
3. 再更新商品库存

如果两个请求几乎同时下单，可能都查到库存足够，然后都扣减成功，产生超卖。

这类问题更适合交给数据库用一条条件更新语句处理。

## 2. 这次怎么修

## 2.1 后端限制订单状态流转

现在订单状态只能按下面规则变化：

```text
普通用户：
待支付 -> 待发货
待支付 -> 已取消
待收货 -> 已完成

管理员：
待发货 -> 待收货
```

其他状态跳转都会被后端拒绝。

涉及文件：

- `springboot/src/main/java/com/example/service/OrdersService.java`
- `springboot/src/main/java/com/example/controller/OrdersController.java`

核心思路是：

- 控制器只负责拿当前登录用户、校验订单归属
- 服务层根据当前订单状态、目标状态、当前用户角色判断是否允许流转
- 不再相信前端传来的完整订单对象

## 2.2 下单扣库存改成数据库条件更新

这次新增了数据库层面的库存扣减：

```sql
update goods
set store = store - #{num}
where id = #{goodsId}
  and store >= #{num}
```

如果库存不足，这条 SQL 的影响行数就是 0，后端直接返回“商品库存不足”。

这样即使并发请求同时进来，数据库也能保证不会把库存扣成负数。

涉及文件：

- `springboot/src/main/java/com/example/mapper/GoodsMapper.java`
- `springboot/src/main/java/com/example/service/GoodsService.java`
- `springboot/src/main/java/com/example/service/OrdersService.java`

## 2.3 取消订单返还库存

订单从 `待支付` 变成 `已取消` 时，会把订单里的购买数量返还给对应商品。

注意，这里使用的是数据库中已有订单的 `goodsId` 和 `num`，不是前端传来的值。

这点很重要，因为前端传来的业务关键字段都不能完全信任。

## 2.4 前端订单更新只提交必要字段

原来前端会把整行订单对象传给后端。

现在改成只提交：

```js
{ id, status }
```

这样可以减少前端误传字段，也更符合后端“只允许改状态”的设计。

涉及文件：

- `vue/src/views/manager/Orders.vue`

## 2.5 前端增加路由守卫

现在前端路由会先做基础判断：

- 未登录访问后台页面，跳转到 `/login`
- 普通用户访问管理员页面，跳转到 `/home`
- 管理员访问用户购买页 `/buy`，跳转到 `/home`

涉及文件：

- `vue/src/router/index.js`

前端路由守卫不是安全边界，真正的权限仍然在后端。但它能让用户体验更顺，不会直接进入一个无权限页面再看到一堆接口报错。

## 4. 测试中发现的小细节

PowerShell 直接发送中文 JSON 时，有时会出现编码不稳定，导致后端收到的中文状态值不匹配。

所以这次接口测试最后改用 Node 发送 UTF-8 请求，并使用 Unicode 转义表示中文状态，例如：

```js
status: '\u5f85\u53d1\u8d27' // 待发货
status: '\u5df2\u53d6\u6d88' // 已取消
```

这不是业务代码的问题，而是测试命令在 Windows 控制台里的编码问题。浏览器前端正常会按 UTF-8 发送请求。

## 5. 后续还能继续优化什么

这次已经把订单核心链路稳住了。后面如果继续做，可以考虑：

1. 把订单状态改成枚举或常量类，减少中文字符串散落在代码里
2. 给订单状态流转补单元测试
3. 把错误码从业务 JSON code 逐步升级为标准 HTTP 状态码
4. 给商品、订单、用户表增加更完整的后端参数校验
5. 管理员删除商品时检查是否有关联订单，避免历史订单数据断裂
