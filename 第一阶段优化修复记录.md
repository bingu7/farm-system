# 第一阶段优化修复记录

本记录对应 `优化计划.md` 的“第一阶段：小范围确定性修复”。

修复目标：

- 清理前端已确认的小 Bug 和无用状态。
- 将管理员统计页的全表加载计数改为数据库计数。
- 在订单 Service 层补充归属校验和删除状态限制。
- 收敛文件下载目录，避免继续从上级 `../files` 目录搜索文件。

## 1. 前端已知 Bug 清理

### 1.1 修改密码确认字段拼写

涉及文件：

- `vue/src/views/manager/Password.vue`

问题：

- 页面字段名写成了 `confirmPasword`，少了一个 `s`。
- 虽然同文件内比较逻辑也使用了同一个错误拼写，短期不一定报错，但字段名不规范，后续加表单校验或维护时容易引入误用。

解决方法：

- 将模板绑定改为 `data.user.confirmPassword`。
- 将保存时的新密码确认比较同步改为 `data.user.confirmPassword`。

验证方式：

- 前端构建通过。
- 修改密码页面仍使用同一个字段比较两次输入。

### 1.2 订单页死代码清理

涉及文件：

- `vue/src/views/manager/Orders.vue`

问题：

- `formVisible`、`form` 在订单页没有实际表单使用。
- 状态更新成功后仍设置 `data.formVisible = false`，属于遗留逻辑。
- 查询注释提到前端强制传用户 ID，但当前后端已经根据 token 中用户身份过滤普通用户订单，注释和实际代码不一致。

解决方法：

- 移除未使用的 `formVisible`、`form`。
- 移除状态更新成功后的无效赋值。
- 删除误导性注释，保留简单分页查询参数。

验证方式：

- 前端构建通过。
- 订单列表、查询、状态更新、删除按钮逻辑不改变。

### 1.3 首页无用用户状态清理

涉及文件：

- `vue/src/views/manager/Home.vue`

问题：

- `data.user` 从 localStorage 读取后没有使用。

解决方法：

- 移除未使用的 `data.user`。

验证方式：

- 首页公告列表请求逻辑不改变。

## 2. 统计接口性能修复

涉及文件：

- `springboot/src/main/java/com/example/service/StatisticsService.java`
- `springboot/src/main/java/com/example/mapper/UserMapper.java`
- `springboot/src/main/java/com/example/mapper/GoodsMapper.java`
- `springboot/src/main/java/com/example/mapper/OrdersMapper.java`

问题：

- 管理员统计卡片使用 `selectAll(null).size()` 获取用户数、商品数、订单数。
- 这种写法会把整表数据加载到 JVM，只为了计算数量，数据量变大后会浪费内存和查询时间。

解决方法：

- 在 `UserMapper`、`GoodsMapper`、`OrdersMapper` 新增 `countAll()`。
- 使用 `select count(*) from ...` 在数据库侧完成计数。
- `StatisticsService` 改为调用 `countAll()`。

验证方式：

- 后端编译通过。
- 管理员统计卡片仍返回全站用户数、商品数、订单数。

## 3. 订单权限和删除规则补强

涉及文件：

- `springboot/src/main/java/com/example/controller/OrdersController.java`
- `springboot/src/main/java/com/example/service/OrdersService.java`

问题：

- `updateStatus` 的归属校验主要依赖 Controller 层，Service 层缺少防线。
- `deleteById` 原来直接删除订单，缺少订单存在性、归属和状态检查。
- 如果删除待支付、待发货、待收货等进行中订单，容易造成订单状态和库存恢复规则不清晰。

解决方法：

- `OrdersController.deleteById` 获取当前登录用户并传入 Service。
- `OrdersService.deleteById` 增加事务、订单存在性检查、归属校验和状态限制。
- `OrdersService.updateStatus` 在状态流转校验前增加 Service 层归属校验。
- 普通用户只能操作自己的订单。
- 普通用户只允许删除自己的已取消订单。
- 管理员允许删除已取消或已完成订单。
- 待支付、待发货、待收货等进行中订单不允许直接删除，避免库存错账。

验证方式：

- 后端编译通过。
- 普通用户操作他人订单时返回无权限。
- 进行中订单删除会被拒绝。
- 已取消或已完成订单按角色规则允许删除。

## 4. 文件下载目录收敛

涉及文件：

- `springboot/src/main/java/com/example/controller/FileController.java`

问题：

- 下载接口会依次搜索：
  - `file.upload-dir`
  - 当前工作目录下的 `files`
  - 上级目录的 `../files`
- 其中上级目录搜索范围过宽，不利于控制下载目录边界。

解决方法：

- `resolveDownloadPath` 只从 `file.upload-dir` 指向的目录读取文件。
- 保留扩展名校验和路径归一化校验。
- 暂不强制给 `/files/download` 加登录鉴权，因为商品图片和头像当前依赖该公开路径展示。

验证方式：

- 后端编译通过。
- 上传后的图片仍从配置上传目录下载。
- 非法扩展名仍返回 404。

## 6. 后续注意事项

- 如果后续希望普通用户在页面上删除自己的已取消订单，需要再调整 `Orders.vue` 的删除按钮显示规则。
- 如果历史图片仍存放在旧的根目录 `files/`，需要先迁移到 `file.upload-dir`，否则收敛下载目录后旧图片不会被找到。
- `/files/download` 是否需要登录鉴权，应在区分“公开商品图”和“私有文件”之后单独处理。
- 数据库字段类型和索引优化仍应按迁移脚本单独处理，不应混在本阶段。
