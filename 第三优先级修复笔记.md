# 第三优先级修复笔记

本次修复两个点：删除逻辑保护、本地配置文件方案。

## 1. 删除逻辑保护

### 原问题

原来商品、分类、用户删除基本都是硬删除：

```sql
delete from xxx where id = ?
```

这样会有数据断链风险：

- 删除商品后，历史订单里的 `goods_id` 找不到商品信息。
- 删除用户后，历史订单里的 `user_id` 找不到下单人。
- 删除分类后，该分类下已有商品会变成无归属分类。
- 删除商品后，库存流水 `goods_stock` 也可能找不到商品。

### 本次选择的方案

本次先采用“删除前关联检查”，不是软删除。

原因：

- 改动更小，不需要改数据库表结构。
- 不需要给所有查询补 `deleted = 0` 条件。
- 对当前系统更稳，能先阻止误删破坏历史数据。

如果后面要做更完整的业务系统，可以再升级成软删除。

### 修改内容

新增 Mapper 统计方法：

- `OrdersMapper.countByGoodsId(goodsId)`
- `OrdersMapper.countByUserId(userId)`
- `GoodsMapper.countByCategoryId(categoryId)`
- `GoodsStockMapper.countByGoodsId(goodsId)`

删除前增加检查：

- 删除商品前：
  - 如果已有订单，禁止删除。
  - 如果已有库存记录，禁止删除。
- 删除用户前：
  - 如果已有订单，禁止删除。
- 删除分类前：
  - 如果分类下已有商品，禁止删除。

这些情况都会返回 HTTP 400，并给出明确提示。

## 2. 本地配置文件方案

### 原问题

本机数据库密码、Redis 密码、上传目录如果直接写在公开仓库的 `application.yml` 里，不适合长期维护。

### 修改内容

`application.yml` 增加可选导入：

```yaml
spring:
  config:
    import:
      - optional:classpath:application-local.yml
      - optional:file:./application-local.yml
```

含义：

- 如果本地存在 `application-local.yml`，Spring Boot 会自动读取。
- 如果文件不存在，也不会报错，因为用了 `optional`。
- 本地配置会覆盖默认配置。

`.gitignore` 增加：

```gitignore
springboot/application-local.yml
springboot/src/main/resources/application-local.yml
```

这样真实本机配置不会被提交。

新增示例文件：

```text
springboot/src/main/resources/application-local.example.yml
```

示例文件里使用占位密码。以后需要本地配置时，可以复制一份为 `application-local.yml`，然后写自己的 MySQL 密码、Redis 密码、上传目录。

## 3. 本次验证结果

后端编译：

```bash
mvn -q -DskipTests compile
```

结果：通过。

接口实测：

| 场景 | 结果 |
| --- | --- |
| 管理员登录 `admin/admin` | HTTP 200，登录成功 |
| 删除已有订单关联的商品 | HTTP 400，提示“该商品已有订单，不能删除” |
| 删除已有订单关联的用户 | HTTP 400，提示“该用户已有订单，不能删除” |
| 删除已有商品的分类 | HTTP 400，提示“该分类下已有商品，不能删除” |

## 4. 后续可选升级

如果后面要做更完整的数据生命周期管理，可以考虑软删除：

- 表里增加 `deleted` 字段。
- 删除时改成 `update xxx set deleted = 1 where id = ?`。
- 所有查询默认加 `deleted = 0`。
- 管理端可以增加“回收站”或“已停用”视图。

软删除更适合正式业务系统，但改动范围比本次删除保护更大。
